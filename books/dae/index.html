    代数微分方程工具

    现有工具

    工具的分类是什么...

    2016年的文章，总的来说，主要有两种数学建模方式：

    • 建模语言（modelling languages）
      • 特定领域语言（DSL - domain specific languages）
      • 多领域语言（multi domain languages）
        • Modelica
        • Ascend
        • gPROMS
        • GAMS
        • Dymola
        • APMonitor
    • 通用编程语言（general-purpose programming languages）
      • 低级（low-level）第三代：C, C++, Fortan，例如
        • PETSc-a套件的数据结构和可扩展解的Balay
        • SUNDIALS–套件的非线性代数/微分方程求解器
        • 附：低级语言也常用来开发定制化软件，如大规模有限差分和有限元求解器，以现有的高性能计算架构为目标，如通用图形处理单元（GPGPU - general purpose graphics processing units），可编程场门阵列（FPGA - field programmable gate arrays），矢量处理器和数据流引擎（DFE - vector processors and Data Flow engines）
      • 高级（high-level）第四代：Python，例如
        • Assimulo-a中C和Fortan写的多种ODE/DAE求解器高级接口
      • 多范式的（multi-paradigm）数值语言，例如
        • Matlab
        • Mathematica
        • Maple
        • Scilab
        • GNU Octave

    另外，有一些建模工具给模拟引擎提供了Python接口，例如

    • APMonitor
    • JModelica
    • OpenModelica

    然而，这些API被限制在导入模型、执行仿真和处理结果这三个步骤。

    通用语言和功能性

    特定领域语言是解决特定领域问题而被创建的，多领域建模语言（如Modelica和gPROMS）则更加多功能性，然而，它们也有一些不足：

    • 对操作系统的访问是有限制的
    • 对第三方数值库的访问是有限制的
    • 它们无法拥有全特征编程语言写脚本那样的能力

    通用语言则不会有上述限制，通用语言没有领域特征。

    数据互操作

    求解仿真和优化问题的重要任务有模型配置、仿真设立、仿真执行、数值求解系统微分/代数方程、处理结果。每个任务或许都需要调用第三方库，为了保持数据在模型和第三方库之间的可互操作性，一方面第三方库不能对模型有额外的预处理过程，另一方面专有模型需要能够以通用数据结构的模式被第三方库使用。在仿真执行阶段，很多时候不能在一开始就将模型定义完毕，而是在运行时利用其他软件/计算的结果来定义。通常情况下，仿真不能被局限于直截了当地对时间积分就完事了，还有用户自定义的步骤和需求，关于这一点，全特征的编程语言可以很好地支持和实现。

    模型互操作

    很多时候需要在不同的仿真器之间对其结果进行比较和对标，需要代码生成和模型互换能力，从而自动生成源代码到目标语言，或者导出模型定义到特定的模型配置语言。将已创建模型的功能暴露给另一个仿真器，需要通过预先定义好的标准接口，例如CAPE-OPEN和FMI - Functional Mock-up Interface。

    发布和部署

    当前IT行业的趋势显示使用云解决方案的需求很高，例如SaaS，PaaS和web应用。

    Modelica和Matlab的局限

    建模语言无法完成所有，例如，Modelica可以通过预处理调用外部库的C语言函数，简单的操作步骤可以有，但是需要嵌入到模型中（），而不是通过分离的独立函数模块。gPROMS也可以定义简单的步骤作为任务，但是只在仿真模式下，自定义的结果处理也需要用户自定义的途径。不支持运行时的模型生成复杂操作步骤，也无法调用其他软件，或者需要额外的应用层。Python和Matlab可以访问许多科学软件库，也支持运行时模型生成、完整灵活的复杂操作步骤和结果处理。然而，Matlab是过程型的，缺少面向对象特性和三类环境下的基本建模概念，这使得构建复杂模型或者复杂层级时很困难。

    所以，新的混合方法需要结合建模语言和通用编程语言的优点。

    混合方法由通用语言实现、尽可能类似建模语言语法、高级语言可以访问操作系统的优势、低级语言拥有大量求解数值问题的数值库。总之，一方面有建模语言的语法以提供关键的建模概念，另一方面拥有通用语言的强大能力。

    新特性和编程范式

    系统可以是块状的/分布式的、稳态的/动态的，事件驱动系统的元素（如非连续方程、状态转移网络和离散事件）。通过操作符重载技术（operator overloading technique）和修改后的ADOL-C支持自动微分。所有的DAE工具库都基于ANSI/ISO C++。所以，代码是跨平台，Linux和MacOS和Windows都可以用。支持Boost库。在32/64位的x86和arm架构经过测试，可以支持嵌入式系统。面向对象的特性允许分层模型分解（hierarchical model decomposition），方便了模型重用。支持大量的数值求解器。Sundials IDAS的状态变阶（variable-order），使用变系数BDF求解器求解DAE系统和计算敏感度。IPOPT求解器、BONMIN求解器和NLopt求解器用来求解（混合整数）非线性编程问题。一系列直接/迭代和顺序/多线程的稀疏矩阵线性求解器都有接口，例如SuperLU/SuperLU_MT求解器、PARDISO求解器、PARDISO求解器、Trilinos Amesos/AztecOO求解器。

    DAE工具不是建模语言，也不是集成软件的套件，而是一个更高层次的结构，一个由相互依赖的软件组件组成的架构设计，以提供API用于：

    • 模型开发/配置
    • 模型动作，例如仿真、优化、参数估计
    • 处理结果
    • 生成报告
    • 代码生成
    • 模型交换

    很容易集成到软件套件中，并且带上GUI、嵌入到另一个软件中，或者开一个web service。

    虽然新的混合方法结合了建模语言和通用编程语言的优点，但是都是有代价的。无论是编译器还是解释器（lexical parser和abstract syntax tree）都需在开发时搞定AST处理、错误处理、语法模糊和隐藏问题等。对用户来说，在建模语言中集成别的组件，需要学习建模语言和另一套语言的语法，很费事...

    当下有三种建模技术：

    • 顺序化模块式（sequential modular）
    • 同步化模块式（simultaneous modular）
    • 非因果的基于公式（equation-based）

    基于模型和顺序模块的对比有文章[xx]，基于公式的overview有文章[xx]。

    在基于公式的方法中，所有组成模型、表达过程的公式和变量都会被生成和集中到一起，然后，使用算法同时求解所有方程。方程是以隐式形式出现的，作为状态变量、导数自由度的函数。
