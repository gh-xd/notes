    基于模型的系统工程

    Harmony SE

    V模型的集成系统软件开发流程：需求分析 > 系统分析/设计 > 软件分析/设计 > 实现和单元测试 > 模块测试 > 子系统测试，包含模型/数据/测试数据仓库

    目标：识别系统功能性、识别相关的系统状态和模式、分配系统功能/模式到物理架构

    3个顶层流程元素：需求分析、系统功能分析、架构设计

    HarmonySE使用服务请求驱动 (service request-drive) 的方式：我的理解是基于模式的开发和操作合约 (operation contract)

    例子：

    Requirement analysis -> use case diagram

    • 需求足够清楚时，它们被组织成use cases，制作用例图use case diagram

    System functional analysis -> black box system model with validated/verified operational contracts (代表了功能需求)

    • 首先，使用strcuture diagram 制作用例情景图 use case context diagram，每个block代表了系统参与者actors
    • 其次，识别use case scenarios。scenario描述了use case中的一种路径（例如，指挥官视角，士兵视角），即在参与者actors/use cases之间的信息流message flow，以及通过操作合约operational contract导致的在接收者recipient处的行为表现。SysML中，scenarios通过制作顺序图sequence diagram来表达。
    • 当scenarios被捕获，功能流functional flow的信息会被合并到一个共同的use case描述中，即生成动作图activity diagram，其中每一个块block就是一个顺序图中的一个操作合约，对接下来的架构设计architectural design非常重要。
    • 下一个分析use case的步骤是描述系统级的、基于状态的行为，使用生成的状态图statechart diagram，是一个分层状态机，描述系统状态/模式以及它们应对外界刺激的变化，基于状态的行为是从use case scenarios中推演出来的。
    • VV验证对象是用例模型use case model，VV验证的开展是执行模型中的用例情景use case scenarios。
    • 注意：这里主要是分析sequence，而非functionality

    Architectural design -> system architecture model, deployment model

    • 目的是配置验证过的操作合约到物理架构，是与领域专家一起迭代完成的。可能会有不同的架构方案和配置策略。
    • 首先，定义物理子系统physical subsystems，使用的是制作结构图structure diagram，组成元素是actor block和system block。部分系统block将是选中架构的物理子系统。
    • 接下来，使用activity diagram把之前捕获的操作合约分配到物理子系统上。基于选中的design concept，操作合约“移动”到对应的子系统swim lane上
    • 白盒顺序图用于识别子系统的接口。
    • 例子：有6个子系统，第一个子系统结果是指挥官的I/O设备，第二个是...，第5个子系统则需要大部分的功能。
    • 当模型的功能性验证完毕，需要分析architectural design的性能和安全需求，包括失效模式结果分析FEMA，使命重要性分析等

    Subsystem architectural design ->

    • 主要是实现操作合约（例如，是硬件？是通过机械还是ASIC码实现合约？还是通过软件？），需要专家。
    • 定义子系统接口和端口，回到architectural design

    Implementation ->

    • 使用deployment model，给到软硬件设计

    Harmoney SE是一个专为软件打造的流程，主要表现为服务请求驱动的开发方式 -- 模式开发和操作合约

    INCOSE OOSEM

    面向对象的系统工程方法，自顶向下、基于模型来支持规范、分析、设计和系统验证。是一个基于OO的婚房方法。自顶向下有利于拓展和需求变更

    目标：捕获/识别那些规范复杂系统的需求和设计信息、集成软硬件OO和其他方法、知识系统级重用和设计升级

    OOSEM的基础：

    • OOSEM独特：企业模型、因果分析、情境细化、需求变种分析、系统/逻辑分解...
    • OOSEM共通：Top-down SE、递归SE、用例/情景驱动（需求图）、黑盒白盒、UML/SysML

    开发行为，V：

    • 利益相关者分析 -> 导出使命需求/高级用例
      • as-is (enterprise model, causal analysis)
      • to-be reqts ( 导出 mission requirement, top-level use cases)
    • 定义系统需求 -> 黑盒系统功能/接口/数据/性能需求 | 系统需求到top-level用例的跟踪 | 需求变种
      • top-level use case /scenarios use activity diagram and swim lines > 导出黑盒系统功能/接口/数据/性能需求
      • 需求管理数据库在activity期间更新/追踪系统需求到top-level use case和mission requirement。系统变种基于可能的系统变更 > 如何设计系统适应潜在变更
    • 定义逻辑架构 -> 分解的逻辑组件
      • 逻辑组件捕获系统功能性（例如用户接口由web浏览器实现、环境监视由传感器实现）
      • 逻辑架构/组件缓解了需求变更对系统设计的影响
    • 合成潜在的配置好的架构 -> 系统-逻辑-组件关系 | 组件需求到系统需求的跟踪
      • 配置好的架构描述了物理组件之间的关系（硬件、软件、数据、人工步骤）
      • 系统节点定义了资源分配，即，每个逻辑组件首先被映射到系统节点，代表功能性是如何分配的
      • 给逻辑组件分配硬件、软件、数据、人工步骤等
      • 对组件的需求跟踪到系统需求
    • 优化/评估替代方案 > trade-off | 性能、可靠性、可用性等参数
      • 为所有的OOSEM activity调用，做trade study
      • 参数模型：建模性能、可靠性、可用性、生命周期费用及其他工程考虑，可追踪，用于优化候选架构
    • 证实和验证系统
      • 目的：系统设计符合需求、需求符合利益相关者需要
      • 输入：系统级用例、情景、相关需求
      • 处理：构建测试案例和验证步骤
      • 需求管理系统跟踪系统需求、系统验证方法的设计信息、用例和结果等

    OOSEM使用面向对象方法集成其他方法，自顶向下地从利益相关者的现状出发，分析出企业级的使命需求和顶层用例，推导出黑盒的系统功能/接口/数据/性能等需求，根据功能性分发逻辑组件、分配具体的物理组件。而后，基于建好的模型及参数，可以全局地推导出基于不同需求变种所得到的方案，通过设计好的测试案例从而自底向上地验证需求的的满足。期间每一步的需求都被跟踪到。

    模型驱动的系统开发 IBM系统工程统一流程

    MDSD - Model-driven systems development

    RUP SE架构框架

    模型层级 Worker模型视角 Logical模型视角 Information模型视角 Distribution模型视角 Process模型视角 Geometric模型视角
    Context (系统黑盒) 角色定义、activity建模 用例图配置 企业数据视角 领域相关的视角 非领域相关的视角
    Analysis (系统白盒) 区分系统 产品逻辑分解 产品数据概念模式 产品本地视角 产品流程视角 布局
    Design (实现软硬件人级别的分析) 操作者指南 软件组件设计 产品数据模式 电子控制没接设计 Timming图 机械计算机辅助设计
    Implementation (实现设计模型到特定配置)
