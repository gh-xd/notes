    SysML

    所有的SysML图只是一种视图，有如下图：

    • 需求图 Requirement Diagram (req)
    • 结构图（静态视图）
      • 定义块图 Block Definition Diagram (bdd)
      • 内部块图 Internal Block Diagram (idb)
      • 参数图 Parametric Diagram (par)
      • 包图 Package Diagram (pkg)
    • 行为图（动态视图）
      • 用例图 Use Case Diagram (uc)
      • 活动图 Activity Diagram (act)：侧重功能执行顺序、逻辑、角色划分
      • 序列图 Sequence Diagram (sd)：侧重交互，也能描述触发
      • 状态机图 State Machine Diagram (stm)：侧重状态构成和状态切换条件

    1 - SysML需求图

    Requirement diagram

    需求图是SysML的主要介质，传达需求之间的可追溯性，系统模型从需求到结构的可追溯性。

    有7种需求关系，包括需求-需求关联和需求-其他模型元素关联。

    • 复合关系（需求-需求）：包括子需求，复合关系的声明会直接执行所有子需求 —(+)
    • 派生关系（需求-需求）：deriveReqt，下一级的需求，例如是车辆加速->发动机动力 --->
    • 细化关系（需求-其他）：refine，使用用例/活动图等其他元素近一步细化需求 --->
    • 满足关系（任何-需求）：satisfy，描述了模型如何满足需求（并不是指满足了），它只是将需求分配给结构的一种机制 --->
    • 验证关系（testCase等-需求）：verify，表示检验方法的机制 --->
    • 复制关系（需求-需求）：copy，例如产品组的法规要求、合同要求 --->
    • 追溯关系（需求-其他）：trace，用于规范树或者连接源文档等

    2 - SysML块定义图

    在块定义图block definition diagram (BDD) 中可以显示其他模型元素，例如模块、执行者、参数约束等。BDD也会出现在其他8种SysML图中。BDD中的元素叫定义元素。定义元素形成了系统模型中其他内容的基础。

    定义元素的关系（关联、泛化和依赖）非常重要。

    2.1 - 模块

    创建Block等于创建一个类型，名称:类型，模块的注释为<>

    • 结构特性：或者叫属性
      1. 组成部分属性：parts & values，模块是由组成部分属性构成的。硬件领域，所属关系实指物理组成关系；软件领域，所属关系是指一个对象负责创建和销毁另一个对象，当为一个组成部分释放内存，组成部分的内存也都被释放。
      2. 引用属性：references&values，“需要”模块外部的一个结构，或者是提供了一种服务，或者是交换事件、能量或数据。
      3. 值属性：values，可以代表一个类型例如布尔值、字符串，特别是和约束属性结合的时候，可以构建系统的数学模型。
      4. 约束属性：constraints&values，约束关系代表一个数学关系，会使用一系列的值属性。约束属性的类型必须是一个存在于其他地方的约束模块的名称。虽然直接在属性框里制定约束表达式也没问题，仅适用于该约束不需要在别的地方重用。
      5. 端口：通过端口，外部实体可以和该结构进行交互，添加了端口等于是针对它的环境创建了一个黑盒，结构内部会对客户端隐藏。端口表达交互点，可以是硬件物理对象（HDMI接口）、软件对象边界交互（消息队列、数据文件）、商业组织的交互（支付订单、邮箱）。注：SysML v1.2之前为标准端口和流端口，v1.3之后为完整端口和代理端口。
        • 标准端口：提供/请求服务，小方块。标准端口可以拥有一种/多种类型，类型是赋予的接口。接口<>和模块类似也是一个元素，是客户端和提供方需要遵循的契约
          • 提供接口用圆形⭕️
          • 请求接口用半圆
    • 行为特性：模块可以执行一系列行为。操作和接收可以定义在模块上，也可以在端口上，根据设计原则定。
      1. 操作：operation，操作由调用事件触发。常见情况是它同步行为，调用者会一直等待行为结束，而SysML不是，可以把任何行为表示为操作，即使调用者没有等待它完成。如果有返回值，返回值类型需要在其他地方定义好。
      2. 接收：reception，接收是由信号事件触发。信号也是一种模型元素<>，它可以是任意类型的事件、能量或数据，目的是要触发接收端的行为，而且信号也有属性properties。接收与操作不同的是，接收总是一种异步行为。

    2.2 - 模块关系

    模块的关系有：

    • 关联：如结构特性中提到的组成部分属性和引用属性，结构直接的关联分为：

      • 引用关联：BDD中引用关联是一条直线\(——\)，有箭头表示单向访问，无箭头则是双向的。可视化箭头和引用属性等价。

      • 组合关联：BDD中组合关联是一条直线—和一个黑菱形。两个模块的组合关联表示结构上的分解。例如，一个卫星由一个电子电力系统、一个高度及轨道控制子系统、一个环境控制子系统和一个通信和数据子系统组成。

    • 泛化：BDD中泛化是直线\(——\Delta\)。代表两种元素的继承关系。抽象的叫超类，具体的叫子类。泛化在系统中创建类型层级关系（分类树）。可以解读为“<子类>是一种<超类>”，例如“<陀螺仪>是一种<传感器>”。泛化可传递，任意深度。

      • 子类可以拥有超类不具备的特性
    • 依赖：BDD种泛化是\(--->\)。一般创建依赖是为了可跟踪（例如需求图中所有的关系基本都是--->，为的是可追溯）。例如，在元素做出变更时，可以查询模型工具，找到可能产生变更的其他元素。依赖时模型的一部分，但不是模型表示系统的一部分

    2.4 - 值类型

    和模块一样，值类型是一种定义元素，一般会出现在模块的值属性。

    通常可以定义三种值类型：

    • 原始值类型<>：没有任何属性，有四种
      • String
      • Boolean
      • Integer
      • Real
      • 自定义（例如温度开尔文时Real的子类型）
    • 结构值类型<>：等于是结构体，两种或多种值类型
      • SysML定义了Complex类型，包括两个值属性realPart和imaginaryPart，两者都是Real
    • 枚举值类型<>：只定义一系列数值，如果一个操作的类型是枚举，那么它必须是枚举中的值。

    4 - SysML参数图

    Parametric Diagram是一种特殊的内部块图，是SysML图中唯一可以表达系统设计方面的内容，用于说明系统的约束，以数学模型表示。并非一定需要使用。

    它关注的是值属性和约束参数之间的绑定关系。和内部块图互补了。

    参数图涉及的主要元素有：

    • 约束模块 Constraint Block <>，约束表达式在约束中段constraints处显示
    • 约束属性 constraints，有名称和类型
    • 绑定连接器 Binding Connector：绑定的两端一个必须是约束参数，另一个是约束参数/值属性

    5 - SysML包图

    Package diagram

    pkg-包是一种通用分组机制，用于在唯一的命名空间内组织各种模型元素和相关的图，可以包含任意SysML元素。

    模型的组织方式由包的层级关系定义，包的层级关系将模型中的元素分配到逻辑上紧密相关的组中。

    在大型项目中，包图描述系统主要元素之间的关系、编译时分组机制、为元素模型提供命名空间。

    包可以包含其他包(containment)、从其他包引入所有元素到命名空间(Package import)、从其他包引入元素到命名空间(Element import)

    7 - SysML活动图

    系统的动态视图，表达随着时间推移的行为和事件发生的序列。可以表达复杂的控制逻辑。

    唯一一个能说明系统连续行为的图。

    活动图的特点：

    • 说明动作执行顺序
    • 只能有选择地说明哪个结构触发哪个动作
    • 有些方法需要活动图，例如添加新用例图时创建活动图（Harmony SE）

    活动元素

    Activity是活动图中唯一可以创建的元素。它是一种行为，也是命名空间，与模块和包类似，因此可以去包含节点和边。

    令牌流

    可以在活动图之间流动，可以有多个，每个令牌都有自己的类型和状态。

    基本动作

    圆角矩形，action，是原子级的Activity，可执行，用自然语言描述。用边来连接。

    对象节点

    矩形，node，出现在两个action之间，表示第一个action产出对象令牌，第二个action将该对象令牌作为输入。

    栓- 方形，pin，特殊类型的对象节点，附加在action上。

    活动参数 - 在活动图的外框上面，从总体上表示活动的一种输入或者输出

    边

    两种边：

    • 对象流 object flow：连接两个对象节点，传输对象令牌，可以表示事件、能量或数据的实例通过action
    • 控制流 control flow：控制令牌可以达到等待它的动作。

    8 - SysML序列图

    说明系统动态行为，可以用生命线lifelines的元素给行为参与者建模，在生命线之间传递消息，为参与者交互建模，可以给约束。

    序列图在系统声明周期早期非常有用，说明子系统之间的交互；在末期也很重要，说明组件之间的交互。可以替换文字版的测试案例说明。

    生命线上可能出现6种类型的时间：

    • 消息发送事件
    • 消息接收事件
    • 生命线创建事件
    • 生命线销毁事件
    • 行为执行开始事件
    • 行为执行结束事件
